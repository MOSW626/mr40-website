# MR40 사이트 고도화 계획 — 시니어 접근성 · 모바일 · 운영 자립

작성: 2026-06-12 · 상태: 검토 대기

## 1. 전제

- 방문자 연령대가 20대 현역부터 60대 초중반 1기 선배까지 걸쳐 있다. 40년치 동문이 대상이므로 "젊은 사람 기준"으로 만들면 안 된다.
- 유입 경로는 대부분 기수별 단톡방에 공유된 링크다. 즉 첫 화면은 거의 항상 카카오톡 인앱 브라우저에 휴대폰 세로 화면이다.
- 운영진은 개발자가 아니다. 행사 전후로 자잘한 수정(공지, 일정, 후원사 추가)이 계속 생기는데, 그때마다 에이전트나 개발자를 거치면 병목이 된다.

## 5. 운영 자립 — 관리자 기능

목표: "공지 한 줄 바꾸는 데 개발자가 필요 없다."

### A안. 운영 시트 방식 (구글시트 = CMS) — 권장 1단계
이미 주소록·방명록이 시트→CSV 패턴으로 돌아간다. 같은 방식을 운영 데이터로 확장한다.

- "MR40 운영 시트" 하나에 탭을 나눈다: `공지`, `행사정보`, `후원사`, `영상목록`, `설문링크`.
- 사이트는 각 탭의 게시 CSV를 읽는다. config.js의 값은 "시트가 비어 있을 때의 기본값"으로 강등.
- 장점: 운영진이 이미 아는 도구(시트), 휴대폰 구글시트 앱으로도 수정 가능, 권한 관리는 시트 공유 설정 그대로, 별도 인증 구현 없음.
- 단점: 반영까지 캐시 시차(수 분), 시트 게시가 풀리면 기본값으로 동작(이미 onFail 패턴 있음).
- 작업량: common.js에 시트 로더 재사용 + 각 페이지 연동. 1~2일 규모.

구현: 탭별 CSV URL과 로컬 설정/JSON 폴백 방식 완료. (2026-06-12, 실제 시트 URL 입력 대기)

### B안. admin.html + GitHub 토큰 방식 — 2단계
사이트 안에 `admin.html`을 만들고, 관리자가 GitHub fine-grained 토큰(이 리포 contents 쓰기 권한만)을 입력하면 GitHub API로 `data/*.json`을 직접 수정·커밋한다.

- 화면 구성: 행사 정보 / 연혁 항목 / 영상 / 후원사 / 회지 목록을 폼으로 편집 → "저장" 누르면 커밋 → 몇 분 뒤 자동 배포.
- 토큰은 localStorage에만 저장하고 화면에 노출하지 않는다. 토큰 만료 1년, 권한 최소화.
- 장점: 시트보다 구조적인 데이터(연혁·회지)까지 편집 가능, 변경 이력이 git에 남음.
- 단점: 토큰 발급·보관을 운영진에게 가르쳐야 함, 잘못 저장하면 JSON이 깨질 수 있어 저장 전 검증 필수.
- 작업량: 3~5일 규모. JSON 스키마 검증과 미리보기까지 넣어야 안전하다.

### 권장 조합
- 자주 바뀌는 것(공지·행사정보·후원사·영상)은 A안 운영 시트.
- 가끔 바뀌는 구조 데이터(연혁·회지)는 B안 admin 페이지로, 여유가 되면 행사 후에.
- 사진 추가는 당분간 현행 유지(photos/ 폴더 + 스크립트). 빈도가 낮고 용량 문제로 자동화 이득이 작다.

## 7. 추가 콘텐츠·기능 (운영진 요청)

### 7.1 취업·창업 현황 통계
- 데이터 출처는 연락처 업데이트 설문이다. 소속 문항을 "회사명 + 분야(대기업/스타트업 창업/교수·연구/공공/기타)"로 받으면 집계가 바로 된다. 별도 조사를 돌리지 않는다.
- 표현: "숫자로 보는 MR" 페이지(stats.html)에 분야별 분포 막대그래프. 서버 없이 Chart.js(CDN) 또는 CSS 막대로 그린다. 데이터는 운영 시트의 집계 탭에서 읽는다.
- 자랑 코너 "MR에서 시작한 회사들": 동문 창업 기업을 로고·한 줄 소개·연도와 함께 나열. 본인 동의를 받은 항목만 싣는다.
- 개인정보 원칙: 통계는 집계 수치만 공개하고 개인 식별 정보는 싣지 않는다. 창업 기업 코너만 예외로 본인 동의 기반.

### 7.2 부원 통계
- 기수별 인원 추이, 재학생/졸업생 비율, 전공 분포 같은 기본 통계를 같은 stats.html에 담는다.
- 사이트가 이미 가진 데이터로 자동 산출되는 재미 수치도 함께: 보유 사진 수, 영상 수, 회지 권수, 연혁 항목 수, 옛 홈페이지 보존 연도 수. 갤러리·연혁 JSON에서 읽으면 되니 운영 부담이 없다.
- 주소록 시트가 차오르면 "40년간 함께한 동문 N명" 카운터를 메인 히어로에 노출한다.

### 7.3 행사 당일 라이브 방송
- 행사를 YouTube Live로 송출하고, 라이브 주소를 운영 시트의 행사정보 탭에 넣으면 사이트가 읽어 표시한다.
- 표시 위치: 행사 당일이 되면 메인 D-day 카운터 자리가 "지금 라이브로 보기" 버튼으로 바뀐다. event.html에도 라이브 임베드를 건다.
- 참석 못 하는 해외·지방 동문을 위한 기능이므로 행사 1~2주 전 공지에 함께 안내한다.
- 종료 후에는 같은 자리에 다시보기 링크를 남긴다.

### 7.4 참가비 카드 결제
- 방침 확정: 페이앱 결제링크(카드)와 토스 송금링크(이체)를 병행한다. 비영리 사업자등록증으로 페이앱 가입.
- 사이트 작업: event.html과 첫 화면 팝업에 결제 버튼 두 개(카드 결제/간편 송금), 금액표, 환불 규정 표기. 결제자명과 참석 조사 이름을 맞춰 달라는 안내 문구.
- 운영 작업: 페이앱 대시보드와 참석 조사 시트를 주 1회 대조한다. 미납자 리마인드는 단톡방 공지로.

### 7.5 운영을 돕는 기능
- 참석 현황판: 참석 조사 시트를 집계해 "현재까지 N명 참석 예정"을 행사 페이지에 표시. 신청을 망설이는 동문에게 자극이 된다.
- 행사 당일 안내 페이지(day.html): 식순, 오시는 길(지도 딥링크), 주차, 좌석 배치, 비상 연락처를 한 페이지에. 행사장 QR 포스터가 이 페이지로 연결된다.
- 체크인: 참석 조사 시트를 출력하거나 시트 앱으로 현장 체크. 별도 시스템은 만들지 않는다(150명 규모에 과투자).
- 행사 후 사진 모으기: 참석자가 찍은 사진을 받을 구글드라이브 업로드 링크를 행사 후 메인에 띄운다. 모인 사진은 photos/2026으로 들어가 갤러리에 편입된다.
- 후원금 현황: 후원사 섹션에 "모금 현황 N원 / 목표 N원" 게이지를 선택적으로 표시. 투명성이 후원을 부른다. 공개 여부는 운영진 결정.
- 문의 창구: 카카오 오픈채팅 링크를 푸터에 상시 노출. 폼이 어려운 분들의 우회로 역할도 한다.
